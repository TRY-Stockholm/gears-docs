## What you'll learn

How `@made-people/grease-checkout` plugs into a client frontend — which pieces it owns, which pieces `grease-centra` owns, and where your slot components land.

## The four things to know

1. **`grease-checkout` ships no Provider.** It is a contract package. You get slot type contracts (address form, shipping, free order, cart-modified) plus the `checkoutEventEmitter` singleton and `CheckoutEventEmitterContext`. That's it.
2. **`<CentraCheckout>` is the engine.** It lives in `@made-people/grease-centra`. It mounts `CheckoutEventEmitterContext`, the UI config provider, and `CentraCheckoutProvider` (the paymentPages context) internally, then renders `CentraCheckoutFlow` to fill the stages with the slot components you pass via `uiConfig.components`.
3. **Your slots are typed against `grease-checkout`.** The component props (`CheckoutAddressForm`, `CheckoutShippingMethodsComponent`, `CheckoutShippingMethodOptionComponent`, `FreeOrderComponent`) all come from this package.
4. **Events are a singleton.** `checkoutEventEmitter` is module-level. `grease-gtm` already subscribes for GA4 `begin_checkout`, `add_shipping_info`, and `add_payment_info`.

## 1. Prerequisites

- `grease-cart` wired and hydrating (see the Cart guide).
- A commerce backend driver — in practice Centra, via `@made-people/grease-centra`.
- A payment provider configured in Centra: Adyen, Qliro, Klarna, or standalone. The PSP iframe/drop-in is rendered by `grease-centra`.
- A shipping provider — Centra-native methods or Ingrid (`IngridProvider` from `@made-people/grease-ingrid`).
- Peer deps: `@made-people/managed-props ~1.8.1`, `@made-people/react-prop-guard ~1.6.1`, `react@18.2.0`, `react-dom@18.2.0`, `zod@^3.22.4`.

```bash
# Inside the monorepo
rush add -p @made-people/grease-checkout
rush add -p @made-people/grease-centra

# Outside the monorepo
pnpm add @made-people/grease-checkout @made-people/grease-centra
```

## 2. Mount `<CentraCheckout>` on the checkout route

There is no `CheckoutProvider` to add. Place `<CentraCheckout>` inside your Centra and Cart providers and pass slot components through `uiConfig.components`.

`uiConfig.components` is typed as `CentraCheckoutUiConfigComponents` — a dictionary of every slot the checkout flow renders. The common ones you'll always set are listed below; the full list of overridable slots (loading states, error views, zero-total flow) appears in the table at the end of §3.

```tsx
import { CentraCheckout, CentraReviveCart } from "@made-people/grease-centra";
import {
  ShippingMethodOption, ShippingMethods,
  ShippingMethodsNotConfigured, PaymentWrapper,
} from "@made-people/blueprint-checkout";

<CentraReviveCart />
<CentraCheckout
  uiConfig={{ components: {
    AddressForm: StrongerAddressForm,
    ShippingMethods, ShippingMethodOption, ShippingMethodsNotConfigured,
    QliroWrapper: PaymentWrapper, AdyenDropInWrapper: PaymentWrapper,
    KlarnaCheckoutWrapper: PaymentWrapper, IngridWrapper: PaymentWrapper,
    FreeOrder,
  }}}
  paymentPages={{ /* … */ }}
/>
```

> **`grease-checkout` owns slot contracts and the event emitter. `grease-centra` owns flow, PSP wiring, and finalize.**

## 3. The checkout flow

```
Cart (CentraReviveCart hydrates selection)
  └─ Address (AddressForm slot — CheckoutAddressForm)
      └─ Shipping (ShippingMethods + ShippingMethodOption, or IngridWrapper)
          └─ Payment (QliroWrapper / AdyenDropInWrapper / KlarnaCheckoutWrapper, or FreeOrder)
              └─ Finalize (<CentraCheckoutFinalize> at paymentReturnPage)
```

| Stage | Owned by | Slot type |
|---|---|---|
| Cart revive | `grease-centra` | — |
| Address | `grease-checkout` contract | `CheckoutAddressForm` |
| Shipping (Centra) | `grease-checkout` contract | `CheckoutShippingMethodsComponent`, `CheckoutShippingMethodOptionComponent` |
| Shipping (Ingrid) | `grease-centra` renders iframe | `IngridWrapper` (layout only) |
| Payment | `grease-centra` renders PSP | `QliroWrapper` / `AdyenDropInWrapper` / `KlarnaCheckoutWrapper` (layout only) |
| Zero-total | `grease-checkout` contract | `FreeOrderComponent` |
| Finalize | `grease-centra` | `GenericOrderReceipt`, `SelectionNotFound` |

`CentraCheckoutUiConfigComponents` also exposes additional slots you can override: `StandalonePaymentWrapper`, `CartIsEmpty`, `CheckoutLoading`, `PaymentLoading`, `UnsupportedCheckoutFlow`, `UnsupportedPaymentMethod`, `UnsupportedShippingMethod`, `PaymentMethodFailed`, `CentraShippingMethodsRenderer`, `CheckoutPaymentGate`, and `YourCartHasBeenModified`. Defaults are provided, so only override the ones you need to brand.

## 4. Events on `checkoutEventEmitter`

| Event | Payload |
|---|---|
| `checkoutStarted` | `{ cart } & CheckoutEventAdditionalData` |
| `shippingMethodSelected` | `{ cart } & CheckoutEventAdditionalData` |
| `paymentMethodSelected` | `{ cart } & CheckoutEventAdditionalData` |

Pass a custom emitter to `<CentraCheckout checkoutEventEmitter={…}>` for test isolation.

## 5. Address form

Build your form against `CheckoutAddressFormProps`:

```tsx
import type {
  CheckoutAddressForm, OnUpdateFunction,
  ValidateAddressFunction,
} from "@made-people/grease-checkout";

export const StrongerAddressForm: CheckoutAddressForm = ({
  address, shippingAddress, validateAddress, onUpdate,
}) => {
  // render fields from address.email, .firstName, .country…
  // call validateAddress({ address, shippingAddress })
  // call onUpdate({ address, shippingAddress })
};
```

`CheckoutAddress` fields carry `{ value, required, validationResult }`. `country` is a select with `options: AddressCountry[]` and optional `states`.

`validateAddress({ address, shippingAddress })` returns `CheckoutAddresses | true` — `true` means the address passed validation, otherwise you receive the addresses back with `validationResult` populated per field. Handle the result before calling `onUpdate`.

To react to updates (e.g. identify a contact in Voyado Engage), wire it at the flow level — not on the slot:

```tsx
flow={{ singlePaymentMethodFlow: { checkoutAddressForm: {
  onAddressUpdate: ({ address }) => { /* … */ },
}}}}
```

`flow.singlePaymentMethodFlow` also accepts `paymentInitialize`, `beforeContent`, and `afterContent` hooks for injecting logic or markup around the payment stage. If you need to block progress until something resolves (e.g. terms acceptance), use `flow.singlePaymentMethodFlowWithBlocking` instead — same shape, with gating support.

## 6. Shipping and payment wrappers

For Centra-native shipping, implement `CheckoutShippingMethodsComponent` (a `PropsWithChildren` wrapper) and `CheckoutShippingMethodOptionComponent` (radio-like; receives `label`, `methodKey`, `price`, `priceNumber`, `isSelected`, `onChange`, `loading`, `isFreeShipping`).

For Ingrid, mount `IngridProvider` in your bootstrap and pass a layout wrapper as `IngridWrapper`. `grease-centra` renders the iframe inside it.

Payment wrappers are `FunctionComponent<PropsWithChildren>` — layout only. The PSP drop-in, iframe, and redirect all live in `grease-centra`. For zero-total carts, the `FreeOrder` slot (`FreeOrderComponent` — `{ placeOrder }`) renders instead.

## 7. Finalize

`<CentraCheckoutFinalize>` from `grease-centra` owns the finalize step. `postData` carries payment-method fields collected by a Remix action on redirect PSPs.

```tsx
import { CentraCheckoutFinalize } from "@made-people/grease-centra";

<CentraCheckoutFinalize
  postData={actionData?.paymentMethodFields}
  uiConfig={{ components: {
    GenericOrderReceipt: (props) => <SuccessPage {...props} />,
    SelectionNotFound: () => <CartNotFound />,
  }}}
/>
```

`GenericOrderReceipt` receives the finalized order. `SelectionNotFound` renders when the Centra selection is gone (expired session or double finalize).

## Gotchas

- **No `CheckoutProvider` export.** Mount `<CentraCheckout>` instead — it provides `CheckoutEventEmitterContext` internally.
- **`src/types/checkout.ts` is a stub.** The file is effectively `export {}` — don't look here for the slot contract types. They live alongside their components (e.g. `CheckoutAddressForm` types next to the address form module).
- **`ShippingMethodsNotConfigured` is a `grease-centra` slot**, not a `grease-checkout` contract.
- **`checkoutEventEmitter` is a module-level singleton.** Pass a custom one for test isolation.
- **`OnUpdateFunction`'s type signature is `() => void`** — Centra's implementation is async under the hood (cast through the type), so awaiting it works at runtime even though the type doesn't declare it.
- **`onAddressUpdate` is a flow prop.** It belongs under `flow.singlePaymentMethodFlow.checkoutAddressForm.onAddressUpdate`, not on the `AddressForm` slot.

## Troubleshooting

**`useCheckoutEventEmitter must be used inside a <CheckoutEventContext />`** — a child is calling the hook outside `<CentraCheckout>`. Mount it higher, or pass the emitter explicitly.

**Finalize renders `SelectionNotFound`** — the Centra selection/token was lost during redirect. Check `CentraLocalStorageTokenHandler` TTL and that the return URL preserves cookies/localStorage.

**`onAddressUpdate` never fires** — it is wired under `flow.singlePaymentMethodFlow.checkoutAddressForm`, not on the `AddressForm` slot.
