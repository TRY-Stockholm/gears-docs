## What you'll learn

How to add a runtime guard to components whose props cross an untyped boundary (CMS, BFF, third-party APIs), what `withZod` actually does on failure, and when TypeScript alone is enough.

## 1. The problem it solves

TypeScript catches wrong types at compile time. But at runtime, data crossing boundaries can arrive in shapes TypeScript assumed away:

- Storyblok bloks come from a CMS — the editor can add, rename, or remove fields.
- BFF responses wrap Centra / third-party data — a schema change upstream can silently break a consumer.
- A new caller of a shared component may type-cast their way past the public API.

When any of these go sideways, the symptom is usually a mystery render bug deep inside a child component — an `undefined` read in a `.map`, a variant resolver taking the wrong branch, a discriminated union mis-dispatching.

`withZod` adds a validation barrier at the component boundary. Before the component renders, props are parsed against a Zod schema. If parsing fails, the component renders nothing (or a supplied fallback) and the Zod error is logged to the console with the offending props attached — so the problem surfaces at the boundary instead of five frames deeper.

## 2. When to use it

Reach for `withZod` when at least one of these is true:

| Situation | Why |
|---|---|
| Props come from a CMS (Storyblok bloks) | Editors can change blok shapes faster than code. |
| Props come from an external API you don't control | Upstream schema drift is invisible to TypeScript. |
| You're publishing a reusable component API (Blueprint, Grease) | Fast failure protects callers from silent bad states. |
| Props shape a discriminated union that gets dispatched on | A malformed discriminator hits the `default` branch with no useful error. |

Skip it when:

- The component is internal to a package and only called from fully typed code.
- Props are derived inside the same render from already-validated state.
- Perf is critical and the component renders in a hot loop (schemas have a real cost — see section 6).

## 3. The API

`withZod` lives in `@made-people/react-prop-guard`. The package has no hand-written `src/index.ts`; the entry point is barrel-generated by `barrelsby` at build time, so `import { withZod } from "@made-people/react-prop-guard"` re-exports from `src/zod/withZod.tsx`.

```ts
function withZod<P extends z.ZodTypeAny>(
  schemaInput: P | { schema: P; defaultProps: z.infer<P> },
  Component: ComponentType<z.infer<P>>,
  ErrorComponent?: ZodPropGuardErrorComponent<z.infer<P>>,
): FunctionComponent<z.infer<P>>;
```

| Argument | Purpose |
|---|---|
| `schemaInput` | Either a Zod schema or `{ schema, defaultProps }` to merge defaults into incoming props before parsing. |
| `Component` | The component to guard. It's called with the parsed `data`, not the raw props. |
| `ErrorComponent` | Optional fallback rendered when parsing fails. Receives the `SafeParseError` result. |

The returned component has `displayName = "Zod(<OriginalDisplayName>)"`, which shows up in React DevTools.

### Failure behavior

The internal `ZodPropGuard` is what does the work. It runs `schema.safeParse(props)` inside a `useMemo` keyed on the props reference. The failure path is the same in dev and prod — there is no `process.env.NODE_ENV` check:

1. `console.error({ passedProps, zodError })` is logged.
2. If an `ErrorComponent` was supplied, it renders with the `SafeParseError`.
3. Otherwise, the guard returns `null` and the children never render.

If you want silent render and a log-only trail, omit `ErrorComponent`. If you want visible fallback UI, pass one. There is no "warn in dev, throw in prod" mode — the guard never throws.

### `defaultProps` and `propMerger`

When you pass `{ schema, defaultProps }`, incoming props are deep-merged on top of defaults before parsing. The merge uses `@made-people/flatten-object` with a `///` delimiter so nested objects merge key-by-key rather than replacing wholesale. React elements and `Date` instances are skipped by the flattener so they survive the round-trip intact.

### Sibling export: `ZodPropGuard`

For render-prop usage, `ZodPropGuard` is exported directly. Use it when you need the guard inside an existing component rather than wrapping one:

```tsx
<ZodPropGuard schema={MySchema} props={props} ErrorComponent={Fallback}>
  {({ data }) => <div>{data.title}</div>}
</ZodPropGuard>
```

## 4. Example: Storyblok component list

Bloks arrive straight from the Storyblok API. An editor can introduce a new blok type, rename a field, or publish partial content — none of which TypeScript sees. Wrapping the list's entry point means a bad top-level payload fails loudly at one spot instead of exploding deep in a blok renderer.

From `src/frontend/grease/storyblok/src/components/storyblok-component-list.tsx`:

```tsx
import type { SbBlokData } from "@storyblok/js";
import { z } from "zod";
import { withZod } from "@made-people/react-prop-guard";
import { StoryblokComponent } from "./storyblok-component";

export const storyblokComponentListSchema = z.object({
  components: z.custom<SbBlokData>().array(),
});

export type StoryblokComponentListProps = z.infer<
  typeof storyblokComponentListSchema
>;

export const StoryblokComponentList = withZod(
  storyblokComponentListSchema,
  (props: StoryblokComponentListProps) => {
    return (
      <>
        {props.components.map((blok: SbBlokData, index) => (
          <StoryblokComponent
            blok={blok}
            key={`storyblok-component-${index}`}
          />
        ))}
      </>
    );
  },
);
```

Two things to notice:

- The schema, the inferred type, and the component are all co-located. `z.infer<typeof schema>` is the one source of truth for the prop type.
- The schema is deliberately shallow (`z.custom<SbBlokData>().array()`). The guard's job here is to catch the "components isn't an array" class of failure — per-blok validation belongs inside each blok's own component.

## 5. Example: ProductProvider

`ProductProvider` receives a `Product` discriminated union (`"simple" | "with_variants" | "bundle"`) loaded from a BFF response. A malformed payload from upstream would otherwise be caught only by the runtime `switch` statement's `default` branch — after the provider tree has started mounting.

From `src/frontend/grease/product/src/context/product-provider.tsx`:

```tsx
export const productProviderSchema = productViewModelSchema.extend({
  product: z.custom<Product>(),
  relatedProducts: relatedProductSchema.array().optional(),
  children: z.custom<ReactNode>(),
  context: z.custom<ProductProviderContext>(),
  fireOnMount: z.boolean().optional(),
});

export type ProductProviderProps = z.infer<typeof productProviderSchema>;

export const ProductProvider = withZod(
  productProviderSchema,
  (props: ProductProviderProps) => {
    // ...dispatch on props.product.dataModel into the right sub-provider
  },
);

ProductProvider.displayName = "ProductProvider";
```

A few details worth copying:

- The schema `extends` a shared view-model schema (`productViewModelSchema`) instead of redeclaring fields. Reuse Zod schemas the same way you reuse TypeScript types.
- `z.custom<Product>()` sidesteps modelling the full discriminated union in Zod — it accepts anything and lets TypeScript narrow. That's a deliberate trade: the guard catches "the envelope is wrong" but not "dataModel is spelled sIMple". Tighten to `z.discriminatedUnion(...)` only if the shape is worth the maintenance.
- `displayName` is set explicitly after the HOC. Without this line, DevTools would show `Zod(undefined)` because the inline arrow function has no name.

`ProductProvider` is further composed with `withZod`'s sibling pattern in several Blueprint components — for example, `Teaser` is wrapped in both `withManagedProps("Teaser", withZod(TeaserPropsSchema, ...))`. The order matters: `withZod` validates the raw props, then `withManagedProps` handles client overrides.

## 6. Authoring tips

- **One schema, one type, one component.** Declare the schema, derive the type with `z.infer<typeof schema>`, export all three from the same file. If a consumer needs the type without the runtime cost, they import the type; if they need to compose, they import the schema.
- **Keep schemas co-located.** Schemas live next to the component that owns them, not in a shared `schemas/` folder. Splitting them makes drift easy.
- **Don't wrap everything.** Each render runs a `safeParse`. It's memoized on the props reference, so stable props are cheap, but schemas with lots of `.refine`/`.transform` can be expensive when props churn. Profile before wrapping a tight list-item component.
- **Use `z.custom<T>()` for shapes Zod can't cheaply model.** React nodes, class instances, and large discriminated unions are candidates. You're trading strictness for speed — that's fine for the envelope.
- **Set `displayName`.** The HOC sets `Zod(<inner>)`; if your inner component is an arrow function with no name, add `Component.displayName = "..."` after the HOC call so DevTools is usable.
- **Compose in the right order.** When stacking with `withManagedProps` / `withDelimitedProps`, put `withZod` closest to the component so it validates post-managed-props shape.

## Gotchas

- **Silent in production too.** The guard never throws. If you don't pass an `ErrorComponent`, a validation failure renders nothing — easy to miss unless someone is watching the console. Consider passing a visible fallback in dev builds.
- **`console.error`, not `console.warn`.** The log uses `console.error`. CI logs and Sentry breadcrumbs will pick these up; filter accordingly if they add noise.
- **`useMemo` is keyed on the props reference only.** Mutating nested fields on the same props object will not re-run parsing. Return new object references from the parent if you need a re-validate.
- **`z.custom<T>()` accepts anything.** It's a type assertion, not a runtime check. Using it on every field defeats the purpose of the guard — save it for shapes Zod can't practically express.
- **`defaultProps` merge uses a flattener.** Keys containing the literal `///` substring would collide with the flattener delimiter. In practice this is theoretical, but be aware if your props carry arbitrary user strings as keys.
- **No `NODE_ENV` branch.** Some guards behave differently in dev vs prod. This one does not — dev and prod behave identically.

## Troubleshooting

**Console error `"passedProps": {...}, "zodError": ZodError {...}` but I see nothing on screen** — the guard is doing its job. Props failed `safeParse` and no `ErrorComponent` was provided. Read the `zodError.issues` array to see which path failed.

**Component renders `Zod(undefined)` in DevTools** — the wrapped component is an anonymous arrow. Assign `displayName` after the `withZod` call, or name the inner function.

**Props look correct but validation still fails** — check for `undefined` vs missing key. `z.object({ foo: z.string() })` rejects `{ foo: undefined }` unless you add `.optional()`. Also check discriminator spelling on unions.

**Changes to `defaultProps` don't appear** — `defaultProps` is captured on first render via `useMemo([props])`. If you're passing a new `defaultProps` reference expecting a re-merge, rebuild the HOC at module scope instead of inside a render.

**Validation runs every render in a list** — props are a new object each render of the parent. Memoize the props object with `useMemo` in the caller, or accept the cost and ensure the schema is shallow.
