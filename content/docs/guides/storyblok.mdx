## What you'll learn

How to mount `@made-people/grease-storyblok` so server-fetched stories render as React components and the visual editor updates live.

## The four things to know

1. **The package renders, it does not fetch.** Stories come in as `ISbStoryData` from your BFF. There is no API client baked in.
2. **Components register globally.** `registerStoryblokComponents` mutates a module-level map. Import the registration file once for its side effects.
3. **Two providers work together.** `StoryblokComponentProvider` holds the transformer stack that rewrites blok data. `StoryblokStoryProvider` holds the active story and wires up live preview.
4. **Preview turns on when two things are true.** `insideStoryblokPreview` is truthy **and** `previewStoryResolver` is passed. Miss one and the bridge never loads.

## 1. Install

```bash
# Inside the monorepo
rush add -p @made-people/grease-storyblok

# Outside the monorepo
pnpm add @made-people/grease-storyblok
```

## 2. Register components

Map Storyblok technical names to React components. Import the module once for its side effects.

```tsx
import {
  registerStoryblokComponents,
  StoryblokComponentList,
} from "@made-people/grease-storyblok";

registerStoryblokComponents({
  Banner,
  Grid,
  PageHeader,
  ContentGroup: StoryblokComponentList,
  MarketContainer: StoryblokComponentList,
  // …
});
```

Wrap each component with `React.forwardRef`. `addStoryblokComponents` checks `isForwardRef` and logs a warning when a component is missing it; `registerStoryblokComponents` only runs `isValidElementType` and stays quiet, so missing `forwardRef` slips through without a warning there.

> **Never skip `forwardRef`.** Without it, `storyblokEditable` data attributes never reach a real DOM node and the visual editor outline disappears — regardless of which registration function you used.

## 3. Wire the component provider

`StoryblokComponentProvider` holds the transformers that rewrite blok trees before render. Mount it once near the root.

```tsx
const storyblokComponentProviderProps: StoryblokComponentProviderProps = {
  transformers: [
    new StoryblokFieldDelimiterTransformer("_"),
    new StoryblokContentReplaceTransformer(
      (value) => (!value.filename ? undefined : { ...value, src: value.filename }),
      (value) => value.fieldtype === "asset",
    ),
    new StoryblokContentReplaceTransformer(
      (value) => ({ ...value, href: value.url }),
      (value) => value?.fieldtype === "multilink" && value?.linktype === "url",
    ),
    new StoryblokContentReplaceTransformer(
      (value) => value.gearsResolvedRichText,
      (value) => "gearsResolvedRichText" in value,
    ),
    new StoryblokBlockListTransformer("children", insideStoryblokPreview),
    // …
  ],
};
```

The real stack lives in `client-frontends/stronger/app/project/bootstrap.tsx`.

## 4. Render a story in a route

Wrap the page in `StoryblokStoryProvider` with the story the BFF returned. Render its blok array with `StoryblokComponentList`. Read from `useStoryblokStory` so live updates propagate.

The snippet below destructures `{ page, frontendContext }` for readability — the shipping file in Stronger takes a single `props: StoryblokRouteProps` argument and reads `props.page` / `props.frontendContext`. Cosmetic only; both forms behave the same.

```tsx
export const StoryblokPageRoute = ({ page, frontendContext }) => {
  const location = useLocation();
  const bffClient = new BffClient({ baseUrl: bffUrl, tenantId });

  const previewStoryResolver = (story: ISbStoryData) =>
    bffClient.resolveStoryblokStory({ story, frontendContext }) as Promise<ISbStoryData>;

  return (
    <StoryblokStoryProvider
      story={page.storyblokData as ISbStoryData}
      insideStoryblokPreview={isStoryblokPreviewRequestQuery(new URLSearchParams(location.search))}
      previewStoryResolver={previewStoryResolver}
    >
      <Content />
    </StoryblokStoryProvider>
  );
};

const Content = () => {
  const { story } = useStoryblokStory();
  if (!story?.content?.pageContent) return null;
  return <StoryblokComponentList components={story.content.pageContent} />;
};
```

Real file: `client-frontends/stronger/app/gears/routes/storyblok-page-route.tsx`.

> **Two forms of `insideStoryblokPreview`.** The prop accepts `boolean | ((blok: SbBlokData) => boolean)`. The boolean form is what you pass at the route level — one flag for the whole tree. The predicate form is used at the bootstrap level (inside a `StoryblokBlockListTransformer`) to exclude specific bloks from the preview wrapper, the way Stronger excludes `CentraSearchResultsProvider`, `ElevateSearchResultsProvider`, and `MenuItem`.

## Live preview

When both conditions hold, `StoryblokStoryProvider` auto-mounts `StoryblokPreviewListener`. It loads the Storyblok bridge from `@storyblok/js` and subscribes to three events:

| Event | What happens |
|---|---|
| `input` | Runs your resolver, calls `setStory` with the result, emits `storyUpdated` on `storyblokEventEmitter` |
| `change` | `window.location.reload()` |
| `published` | `window.location.reload()` |

Inside the editor, swap real links for a non-navigating span so clicks stay in the preview frame:

```tsx
const LinkComponent = isInsideStoryblokPreview ? StoryblokPreviewLink : AppLink;
```

## Gotchas

- **No fetcher.** Stories must be fetched elsewhere (stronger uses `BffClient.resolveStoryblokStory`) and handed in as `ISbStoryData`.
- **`registerStoryblokComponents` mutates module-level state.** Importing the registration module runs it. Use `clearStoryblokComponents()` between tests.
- **Preview wraps each blok in `<div style={{display:"contents"}}>`.** Components that `cloneElement` on their children drop those data attributes — pass the predicate form of `insideStoryblokPreview` to opt them out (stronger excludes `CentraSearchResultsProvider`, `ElevateSearchResultsProvider`, `MenuItem`).
- **`isStoryblokPreviewRequestQuery` requires all eight `_storyblok*` params.** Any URL rewrite that strips one silently disables preview.
- **`change` / `published` reload the page.** Unsaved local state in the preview frame is lost on publish.
- **`StoryblokFieldDelimiterTransformer` unflattens every component.** `foo_bar` becomes `{ foo: { bar } }`. Fields starting with the delimiter are skipped.

## Troubleshooting

**Visual editor clicks do nothing** — either the component isn't wrapped in `React.forwardRef`, or `insideStoryblokPreview` returns `false` for that blok. Verify with `useInsideStoryblokPreview()` in a test render.

**Live edits never appear** — you passed `insideStoryblokPreview` but forgot `previewStoryResolver`. Both are required to mount the bridge listener.

**`Component <Foo> doesn't exist`** — the technical name from the Storyblok space doesn't match any key in `registerStoryblokComponents`. Keys are case-sensitive.

**`useStoryblokStory` / `useInsideStoryblokPreview` throw** — these hooks read from `StoryblokStoryProvider`'s context and throw if called outside it. Wrap the component tree in the provider, or only call them in children of your route component.
