## What you'll learn

How to mount `@made-people/grease-account` (v0.16.6) and `@made-people/grease-order` (v0.10.11) in a React client frontend. Login, session restore, profile edits, address book, order history, and the Voyado Engage wrapper that layers on top.

## The five things to know

Before you wire anything, these five facts decide the shape of your code:

1. **`grease-order` ships no Provider.** It is a types + event-emitter package. Orders live on `AccountState.orders` inside `AccountProvider`. Wiring `<OrderProvider />` will fail — it doesn't exist.
2. **Read account state via `useAccountState`, not `useCustomer`.** There is no `useCustomer` hook. `useAccountState()` returns `{ customer, orders }`.
3. **`customer` is tri-state.** `undefined` = not fetched yet. `null` = fetched, no session. A `Customer` object = logged in. Distinguish all three or you'll redirect logged-in users during hydration.
4. **The driver is the integration seam.** `AccountClientDriver` has eight methods. Clients build `CentraAccountClient` from `@made-people/grease-centra`, then optionally wrap it with `VoyadoEngageCentraAccountDriver`.
5. **Reset-password tokens come from the URL.** `?id=…&i=…` are read by the Centra driver's constructor, not by the hook.

## 1. Prerequisites

- A working commerce driver (typically `CentraAccountClient` from `@made-people/grease-centra`). See [Implementing Centra](/docs/guides/centra).

- Peer deps for `grease-account`: `react@18.2.0`, `react-dom@18.2.0`, `@made-people/grease-order@~0.10.11`, `@made-people/grease-product@~1.29.0`, `@made-people/types-e164-phone-number-string@~1.2.1`.
- Peer deps for `grease-order`: `react@18.2.0`, `react-dom@18.2.0`, `@made-people/managed-props@~1.8.1`, `@made-people/react-prop-guard@~1.6.1`, `zod@^3.22.4`.

## 2. The provider tree

One provider — `<AccountProvider />` — stores both customer and orders. The tree looks like this:

```
AccountProvider                         ← holds customer + orders state
  └─ AccountInitialize                  ← internal; calls fetchCustomer on mount
      └─ children (routes, blueprints)
```

From `client-frontends/stronger/app/project/bootstrap.tsx`:

```tsx
import { AccountProvider, AccountProviderProps } from "@made-people/grease-account";

const accountProviderProps: AccountProviderProps = {
  accountClientDriver: voyadoEngageCentraAccountClient,
};
// …
<AccountProvider {...accountProviderProps}>{children}</AccountProvider>;
```

`AccountProvider` mounts an internal `<AccountInitialize />` that calls `fetchCustomer` on mount to restore the session. Defer with `disableFetchOnMount` or `initializeOnMount={false}`.

> **Orders ride along inside `AccountProvider`.** There is no separate `OrderProvider`. If you need orders, call `useAccountFetchOrders` — don't reach for a provider that doesn't exist.

## 3. The driver pattern

Build bottom-up: Centra first, then wrap with Voyado if you use Engage.

```tsx
import {
  CentraAccountClient, CentraAccountMappers,
  centraCheckoutApiCustomerToGreaseCustomer,
  makeCentraCheckoutApiOrderToGreaseOrderMapper,
} from "@made-people/grease-centra";
import { VoyadoEngageCentraAccountDriver } from "@made-people/grease-voyado-engage";

const mappers: CentraAccountMappers = {
  orderMapper: makeCentraCheckoutApiOrderToGreaseOrderMapper(/* … */),
  customerMapper: centraCheckoutApiCustomerToGreaseCustomer,
};
const centraAccountClient = new CentraAccountClient(
  centraCheckoutAPIClient, mappers,
  { resetPasswordUri: "forgot-password", resetPasswordData: { id, i } },
);
const voyadoEngageCentraAccountClient = new VoyadoEngageCentraAccountDriver({
  voyadoEngageClient, centraAccountClient,
  discoveryKeySetter: { setDiscoveryKey: setVoyadoCustomerKey },
});
```

Pass the wrapped client as `accountClientDriver`. Skip the Voyado wrap if you don't use Engage.

## 4. Hooks grouped by purpose

Every `grease-account` hook returns a `useRequester`-style shape (`{ execute, … }`).

### Auth

| Hook | Purpose |
|---|---|
| `useAccountLogin` | Email/password login, returns a discriminated union: `{ success: true }` or `{ success: false, error }` where `error` is `"Account_MissingFields" \| "Account_InvalidEmail" \| "Account_InvalidCredentials" \| "Account_UnknownError"` |
| `useAccountLogout` | Clears `AccountState` |
| `useAccountRegister` | Register a new customer |
| `useAccountSendResetPasswordEmail` | Triggers the reset email |
| `useAccountResetPassword` | Consumes `?id` + `?i` from URL |

### Session / profile

| Hook | Purpose |
|---|---|
| `useAccountState` | Reads `{ customer, orders }` from context |
| `useAccountFetchCustomer` | Manual session refresh |
| `useAccountUpdateCustomer` | Writes profile + address book |

### Orders

| Hook | Purpose |
|---|---|
| `useAccountFetchOrders` | Lazy; call `.execute()` in an effect |

### Lower-level

| Hook | Purpose |
|---|---|
| `useAccountClient` | Raw driver access |
| `useAccountDispatch` | Dispatch reducer actions directly. `AccountActionTypes` exposes `SET_ACCOUNT_DATA`, `SET_ORDERS`, and `CLEAR_ACCOUNT_DATA` |
| `useAccountEventEmitter` | Subscribe to `loginSuccess`/`loginFailure`/`registerSuccess`/`registerFailure`. Each payload is `{ additionalData?, method }` — `method` is required (e.g. `"email"`, `"google"`) |

> **Tip: enriching event payloads.** Wrap your tree in `<AccountEventAdditionalDataProvider additionalData={…} method="email">` (from `account-event-additional-data-provider.tsx`) and `useAccountLogin` / `useAccountRegister` will pull that context in automatically when they emit `loginSuccess` / `registerSuccess` etc.

## 5. Session state as tri-state

```tsx
const { customer } = useAccountState();
if (customer === undefined) return null;              // not fetched yet
if (customer === null) navigate("/account/login");    // fetched, no session
// else: Customer is logged in
```

Same pattern in `client-frontends/stronger/app/gears/routes/account/account-index-page-route.tsx`.

## 6. Profile, address book, and orders

`Customer` (from `@made-people/grease-customer`) carries `firstName`, `lastName`, `email`, `addresses[]`, and `attributes`. The address book is just the array — mutate it and push the whole customer back:

```tsx
const update = useAccountUpdateCustomer();
await update.execute({ customer: { ...customer, firstName: "Ada" } });
```

Orders are lazy. Fetch inside an effect:

```tsx
const orders = useAccountFetchOrders();
useEffect(() => { orders.execute(); }, []);
const { orders: orderList } = useAccountState();
```

Each zod-validated `Order` from `grease-order` carries `orderNumber`, `currencyCode`, `itemQuantity`, `items[]`, `totals`, `status` (`pending | processing | completed | cancelled | refunded`), `date`, `paymentMethod`, `shippingMethod`, `shippingAddress`, `billingAddress`, and `vouchers[]` (required — empty array when none). Only `additionalNotes` and `customer` are optional.

## 7. Customer / order mappers

Backend responses (e.g. Centra's `CentraLoggedIn`) must be mapped into grease shapes before they reach the reducer. `grease-centra` ships a default customer mapper:

```ts
// grease-centra: centra-checkout-api-customer-to-grease-customer.ts
export const centraCheckoutApiCustomerToGreaseCustomer = (
  loggedIn: CentraLoggedIn,
): Customer => ({
  id: loggedIn.customer,
  firstName: loggedIn.firstName,
  lastName: loggedIn.lastName,
  email: loggedIn.email,
  addresses: [
    {
      firstName: loggedIn.firstName,
      lastName: loggedIn.lastName,
      street: loggedIn.address1,
      street2: loggedIn.address2,
      city: loggedIn.city,
      state: loggedIn.state,
      zipCode: loggedIn.zipCode,
      country: loggedIn.country,
      phoneNumber: loggedIn.phoneNumber,
    },
  ],
  attributes: {},
});
```

Note the shape: `id`, `firstName`, `lastName`, and `email` live on the top-level customer. The address sub-object carries its own name, phone, and street/city/zip/country — but **no** `email` field.

Clients compose mappers inline in `bootstrap.tsx` (see stronger's `strongerCentraMappers`). There is no separate `customer-mapper.ts` convention.

## Gotchas

- **`customer === undefined` vs `null`.** Always distinguish before redirecting, or you'll bounce logged-in users during hydration.
- **`grease-order` has no provider.** Wiring `<OrderProvider />` will fail: it doesn't exist.
- **SSR hydration guard.** `AccountInitialize` only fires client-side. The `undefined` check is critical during SSR.
- **Rewrapping the driver.** `VoyadoEngageCentraAccountDriver` wraps `CentraAccountClient`. Swapping the inner driver later won't propagate unless you rebuild the wrapper.
- **Reset-password tokens.** `?id=…&i=…` go into the `CentraAccountClient` constructor's `resetPasswordData`, not into the hook.
- **Login succeeds but state stays empty.** You called the driver directly. Only `useAccountLogin` dispatches `SET_ACCOUNT_DATA`.

## Troubleshooting

**Users logged in but orders never appear** — `useAccountFetchOrders` is lazy. Call `.execute()` inside an effect.

**Login succeeds but state never updates** — you bypassed the hook. Use `useAccountLogin`, not the driver directly.

**Reset-password flow silently fails** — `id` / `i` aren't reaching `CentraAccountClient`'s `resetPasswordData`. The driver has no token to send.
