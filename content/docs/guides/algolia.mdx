## What you'll learn

How to mount `@made-people/grease-algolia` so a client gets instant search, autocomplete, faceted filters, Algolia Recommend, and automatic Insights events.

## The four things to know

1. **`AlgoliaProvider` sits inside `SearchProvider`.** `grease-search` owns search state, facets, and autocomplete. `grease-algolia` plugs its clients into that shell.
2. **Credentials are fetched, not hard-coded.** An async `algoliaClientDataGetter` resolves `{ application_id, search_api_key, index_name }` from the gears algolia service via `useGearsServiceUrl("algolia")`.
3. **Insights mount automatically — but only with a user token.** Set `algoliaConfig.insights.userToken` and `AlgoliaNotifications` subscribes to product, cart, and order event emitters.
4. **`AlgoliaSearchResultsProvider` only renders product searches.** Any other `algoliaTypeId` silently renders nothing.

## 1. Install

Inside the Rush monorepo, add `@made-people/grease-algolia` to your app's `package.json` and run `rush update`. Outside the monorepo:

```bash
pnpm add @made-people/grease-algolia
```

Peer deps: `@made-people/grease-search`, `@made-people/grease-product`, `@made-people/managed-props`, `react`, `react-dom`, `zod`. Also install the Algolia SDKs the consumer constructs: `algoliasearch`, `@algolia/client-insights`, `@algolia/recommend`.

> **Heads up — implicit deps.** `@made-people/grease-cart` and `@made-people/grease-order` are not declared peer dependencies of `grease-algolia`, but `AlgoliaNotifications` subscribes to their event emitters at runtime. Add them explicitly to your consumer app so the emitters are present.

> **SDK version skew.** `@algolia/client-insights` is on the v5 SDK line (`~5.45.0`) while `algoliasearch` and `@algolia/recommend` are still on v4 (`~4.20.0`). The code below uses both lines side by side — that's intentional, not a typo.

> **Import path.** The search client imports from `algoliasearch/lite` (which matches the `SearchClient` type expected by the provider), not from the package root.

Current package version: `1.6.7`.

## 2. Build the client getter

`AlgoliaProvider` takes an async `algoliaClientDataGetter` returning the three Algolia clients plus `indexName`:

```ts
import algoliasearch from "algoliasearch/lite";
import { insightsClient } from "@algolia/client-insights";
import recommend from "@algolia/recommend";

export const getAlgoliaClient = async (language, market, algoliaServiceUrl) => {
  const { application_id, search_api_key, index_name } = await fetchConfig(
    algoliaServiceUrl,
    { market, language },
  );
  // …
  return {
    searchClient: algoliasearch(application_id, search_api_key),
    insightsClient: insightsClient(application_id, search_api_key),
    recommendClient: recommend(application_id, search_api_key),
    indexName: index_name,
  };
};
```

The gears endpoint looks like `/algolia/v2/context/products?market=…&language=…`. `indexName` is fetched — there is no prop to set it directly on `AlgoliaProvider`.

> **No token, no Insights.** If `algoliaConfig.insights?.userToken` is unset, `AlgoliaNotifications` never mounts and zero events are sent. Generate the token before `AlgoliaProvider` renders.

## 3. Wire up the provider stack

```tsx
import {
  AlgoliaProvider,
  AlgoliaAutoCompleteProvider,
  type AlgoliaProviderProps,
} from "@made-people/grease-algolia";

const props: AlgoliaProviderProps = {
  algoliaClientDataGetter: () => getAlgoliaClient(language, market, url),
  algoliaConfig: {
    mappers: { centraShopProduct, algoliaFacetToGreaseFacetValue },
    insights: userToken ? { userToken } : undefined,
  },
};

<AlgoliaProvider {...props}>
  <AlgoliaAutoCompleteProvider>{children}</AlgoliaAutoCompleteProvider>
</AlgoliaProvider>;
```

Mount inside `<SearchProvider>` from `grease-search`. The tree:

```
SearchProvider                       ← from grease-search
  └─ AlgoliaProvider                 ← clients + config
      └─ AlgoliaAutoCompleteProvider ← plugs into grease-search AutoComplete
          └─ children
```

Miss Mary wires this in `client-frontends/miss-mary/app/bootstrap.tsx` via a thin `<AlgoliaBootstrap>` wrapper at `client-frontends/miss-mary/app/services/algolia/algolia-bootstrap.tsx`.

> **Autocomplete defaults.** `AlgoliaAutoCompleteClient` hard-codes `algoliaTypeId: "product"` and `clickAnalytics: true`. Autocomplete only queries the product index, and every autocomplete click is tracked in Insights.

## 4. Render a search page

`AlgoliaSearchResultsProvider` wraps `CachedSearchResultsProvider` from `grease-search`, so any downstream search hook keeps working:

```tsx
import {
  AlgoliaSearchResultsProvider,
  type AlgoliaSearchPayload,
} from "@made-people/grease-algolia";

const payload: AlgoliaSearchPayload = {
  algoliaTypeId: "product",
  searchQuery: q,
  facetGroupConfigs,
};

<AlgoliaSearchResultsProvider payload={payload}>
  <Search searchTerm={q} />
</AlgoliaSearchResultsProvider>;
```

## 5. Recommendations

```tsx
<AlgoliaRecommendations model="related-products" maxRecommendations={8}>
  <RecommendationList />
</AlgoliaRecommendations>;
```

| Model | `objectID` |
|---|---|
| `related-products` | auto-resolved |
| `bought-together` | auto-resolved |
| `looking-similar` | auto-resolved |
| `trending-items` | not needed |
| `trending-facets` | not needed |

All four product-context models auto-resolve `objectID` from `useMaybeProductBase()` or the first cart item when you omit the prop. You only need to pass it explicitly if neither context is available (e.g. a standalone recommendations page).

## Filters and facets

Facet state is owned by `grease-search`. Selections are translated via `searchResultFacetsToAlgoliaFacetFilters` and applied through `performDisjunctiveSearch`. Pass `facetGroupConfigs` on the payload for grouped facets. Set `algoliaConfig.mappers.resolveFacetValueForFilter` when display IDs differ from Algolia index values (e.g. JSON-encoded color swatches).

## Gotchas

- **Wrong `algoliaTypeId`.** `AlgoliaSearchResultsProvider` silently renders nothing unless `algoliaTypeId === "product"`.
- **Config fetch failure.** `AlgoliaClientDataGetter` may return `undefined`; search is silently disabled.
- **Missing user token.** No `insights.userToken` means no `AlgoliaNotifications`, no Insights events.
- **Missing `objectID` on a non-PDP page.** Only `trending-items` and `trending-facets` work without product context; the other three models rely on `useMaybeProductBase()` or the first cart item to auto-resolve `objectID`.
- **Facet value mismatch.** Without `resolveFacetValueForFilter`, display IDs must match Algolia values exactly.
- **Non-Centra indexes.** `mappers.centraShopProduct` expects Centra Shop API records; supply your own mapper otherwise.

## Troubleshooting

**Search returns nothing** — the `algoliaClientDataGetter` likely resolved to `undefined`. Verify the gears algolia endpoint and credentials.

**Insights not sent** — `algoliaConfig.insights.userToken` is missing. Generate and persist a token before rendering `AlgoliaProvider`.

**Recommendations render nothing** — `AlgoliaRecommendations` returns `null` both while results are still loading (`results === undefined`) and when the result set is empty. If you need a loading skeleton or an empty state, render it outside `AlgoliaRecommendations`. Also double-check that `objectID` could be auto-resolved (or pass one explicitly).

**Facet value mismatch** — supply `resolveFacetValueForFilter` to translate display IDs into Algolia index values.
