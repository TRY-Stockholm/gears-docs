## What you'll learn

How Blueprint components register themselves under a name, how clients fill in default props once at the root, and how GrowthBook flags reshape the same components at runtime — all without forking components or prop-drilling per tenant.

## The three things to know

1. **Components opt in by name.** `withManagedProps("ProductCard", Component)` registers a component under a string key. Anything above it in the tree can reach in and override its props.
2. **Overrides come from a context, not explicit props.** `ManagedPropsProvider` carries a `fallbackProps` / `overrideProps` / `deferredOverrideProps` tree. At render time each registered component merges: `fallbackProps` → explicit props → `overrideProps` → `deferredOverrideProps` (last wins).
3. **GrowthBook is just another override source.** `GreaseGrowthbookManagedPropsProvider` translates feature flag keys shaped `ComponentName::nested_prop` into the same override tree. Clients get theming from `fallbackProps`; experiments get runtime reshape from flags.

## 1. The problem it solves

Blueprint is a multi-tenant component library. Stronger, Miss Mary, and Dedicated all render the same `ProductCard`, `Topbar`, `Heading`, and `Button` — but each needs different defaults (badge colours, quickshop layout, topbar template) and each runs different GrowthBook experiments on top.

The naive options both hurt:

- **Forking components per client** — doubles the maintenance surface and defeats the point of Blueprint.
- **Per-client prop drilling from the page root** — every page has to know every component's shape, and feature flags have nowhere to plug in without touching page code.

Managed-props is the runtime layer that resolves both. A component registers under a name. A provider up the tree supplies the default shape. Feature flags inject overrides against the same name. The component itself never changes.

## 2. The mental model

Each registered component runs this sequence on every render:

```
explicit props from the parent
  ← merged over fallbackProps (client defaults)
  ← merged under overrideProps (static overrides)
  ← merged under deferredOverrideProps (GrowthBook, async sources)
```

The merge is deep. Props are flattened with `@made-people/flatten-object` using an internal `///` delimiter, then re-merged key-by-key, so nested objects like `badges.discountBadge.variants.color` can be overridden without clobbering neighbouring keys. React elements and `Date` instances are skipped during flattening so they stay intact.

Two ways to supply overrides to the provider:

| Shape | When to use |
|---|---|
| Object tree — `{ ProductCard: { badges: { location: "gallery" } } }` | Client defaults written by hand in the codebase |
| Flat string keys — `ProductCard::badges_location` → `"gallery"` | Feature flag tools (GrowthBook) that only support scalar values |

The `GreaseGrowthbookManagedPropsProvider` does the string-to-tree conversion for you.

## 3. The core API

`@made-people/managed-props` has no hand-written `src/index.ts` — the barrel is generated by `barrelsby` during build. Everything listed below comes from `src/` files and re-exports cleanly.

| Export | What it does |
|---|---|
| `withManagedProps(displayName, Component)` | HOC. Registers a component under `displayName` and merges provider-supplied props at render time. |
| `ManagedPropsProvider` | Context provider. Accepts `value: ManagedPropsContextValue`. |
| `ManagedPropsContextValue` | Shape: `fallbackProps?`, `overrideProps?`, `deferredOverrideProps?`. Each is a map keyed by component name, with values that are props objects or factories returning them. |
| `ManagedPropsLiteral` | Type for `fallbackProps` / `overrideProps`. |
| `ManagedDeferredPropsLiteral` | Like `ManagedPropsLiteral` but values may be `null` to signal "still resolving, don't render yet". |
| `useManagedProps(type, componentName)` | Low-level hook used inside the HOC. Rarely needed in application code. |
| `withDelimitedProps(delimiter, Component)` | Secondary HOC that un-flattens already-delimited props (e.g. from a CMS). Strips types — pair with `withZod`. |

The typical Blueprint wrap looks like this:

```tsx
export const ProductCard = withManagedProps(
  "ProductCard",
  withZod(productCardPropsSchema, (props) => {
    const templateProps = props.templateProps || { template: defaultTemplate };
    return <SelectTemplate {...templateProps} />;
  }),
);
```

`withManagedProps` on the outside, `withZod` on the inside. Zod validates the merged final shape, which means a bad feature-flag value won't crash the tree — it gets caught by the schema.

## 4. Example: a registered Blueprint component

`src/frontend/blueprint/product-card/src/ProductCard.tsx`:

```tsx
import { withManagedProps } from "@made-people/managed-props";
import { z } from "zod";
import { withZod } from "@made-people/react-prop-guard";
import { SelectTemplate, productCardSchema } from "./TemplateSelector";

export const productCardPropsSchema = z.object({
  templateProps: productCardSchema.optional(),
});

export type ProductCardProps = z.infer<typeof productCardPropsSchema>;

export const ProductCard = withManagedProps(
  "ProductCard",
  withZod(productCardPropsSchema, (props) => {
    const templateProps = props.templateProps || { template: defaultTemplate };
    return <SelectTemplate {...templateProps} />;
  }),
);
```

The consumer can render `<ProductCard />` with no props and get the template + defaults from the provider. Or it can render `<ProductCard templateProps={{ template: "1", badges: {...} }} />` and those explicit props sit in the middle of the merge.

Client side, Dedicated supplies defaults in `client-frontends/dedicated/app/project/blueprint/components/product-card/default-props.tsx`:

```tsx
export const ProductCardTemplate1 = {
  template: "1",
  productInfoText: { dataKey: "shortDescription" },
  badges: {
    location: "gallery",
    discountBadge: {
      enabled: true,
      variants: { color: "red" },
    },
  },
  quickshop: { /* ... */ },
};
```

These get aggregated into a single `defaultProps: ManagedPropsLiteral` in `client-frontends/dedicated/app/project/default-props.tsx` and handed to `ManagedPropsProvider` in the bootstrap stack.

The same pattern covers every Blueprint component — `Topbar`, `Link`, `Heading`, `Button`, `ProductCard`, all wrapped the same way. Stronger's `client-frontends/stronger/app/project/blueprint/components/topbar/default-props.tsx` is as small as:

```tsx
import { TopbarProps } from "@made-people/blueprint-topbar";

export const Topbar: TopbarProps = {
  settings: { template: "ticker" },
};
```

That's all it takes to skin a component per client.

## 5. Client defaults — the aggregation pattern

Every live client follows the same three-layer structure to build `defaultProps`:

```
blueprint/components/<component>/default-props.tsx   (one file per component)
         ↓ imported by
project/default-props.tsx                             (top-level aggregation)
         ↓ passed as `fallbackProps` to
GreaseGrowthbookProvider / ManagedPropsProvider       (in bootstrap.tsx)
```

The three live clients all follow this shape, with slight variations in directory layout.

| Client | Per-component defaults live in | Aggregated at |
|---|---|---|
| Stronger | `app/project/blueprint/components/<name>/default-props.tsx` | `app/project/default-props.tsx` (~170 lines, 70 entries, 59 files) |
| Miss Mary | `app/components/blueprint/<name>/default-props.tsx` | `app/config/default-props.tsx` (~100 lines, 42 entries, 42 files) |
| Dedicated | `app/project/blueprint/components/<name>/default-props.tsx` | `app/project/default-props.tsx` (~115 lines, 43 entries, 43 files) |

### The top-level aggregate

`client-frontends/stronger/app/project/default-props.tsx`, abridged to the shape:

```tsx
import { ManagedPropsLiteral } from "@made-people/managed-props";
import { Banner } from "./blueprint/components/banner/default-props";
import { Topbar } from "./blueprint/components/topbar/default-props";
import {
  ProductCard,
  ProductCardQuickshopButtonGrid,
  QuickshopDrawer,
} from "./blueprint/components/product-card/default-props";
// ... 60+ more imports

export const defaultProps: ManagedPropsLiteral = {
  Banner,
  Topbar,
  ProductCard,
  ProductCardQuickshopButtonGrid,
  QuickshopDrawer,
  // ... every component name → default-props object
  // Renames are allowed:
  MiniCartSwipeableProductList: MiniCartSwipeableProductListDefaults,
  FabricCard: FabricCardDefaults,
  CollectionCard: CollectionCardDefaults,
};
```

The object is keyed by `displayName` — the string you passed to `withManagedProps`. An imported symbol called `Defaults` gets renamed to the component name in the object literal. One name mismatch silently drops that component's defaults.

### A single per-component file

`client-frontends/stronger/app/project/blueprint/components/topbar/default-props.tsx`:

```tsx
import { TopbarProps } from "@made-people/blueprint-topbar";

export const Topbar: TopbarProps = {
  settings: { template: "ticker" },
};
```

Typed against the component's own `Props` type — so autocomplete works and you can't put `settings.tmeplate` by mistake. The exported name (`Topbar`) matches the key in the aggregate — so spread-import works.

More elaborate components carry more defaults:

```tsx
export const ProductCard: ProductCardProps = {
  templateProps: {
    template: "1",
    productInfoText: { dataKey: "shortDescription" },
    badges: {
      location: "gallery",
      discountBadge: { enabled: true, variants: { color: "red" } },
    },
    quickshop: { /* ... */ },
  },
};
```

Same file shape, richer object. No client-level indirection — the file is what `ManagedPropsProvider` will hand to the ProductCard at render time.

### Wiring it into the bootstrap

Stronger wraps `ManagedPropsProvider` via `GreaseGrowthbookProvider` (which composes both sources — defaults + flags):

```tsx
import { defaultProps } from "./default-props";

<GreaseGrowthbookProvider
  clientProviderProps={{ client, viewedExperiments: props.gbState?.e }}
  managedProps={{ fallbackProps: defaultProps }}
>
  {children}
</GreaseGrowthbookProvider>
```

`fallbackProps` is the layer *below* explicit props — the "I got nothing from the parent, use client defaults" fallback. Feature flags arrive via `deferredOverrideProps` in the same provider. Static overrides (rare; usually debugging) go in `overrideProps`.

### Why one file per component

The fan-out pattern (one defaults file per component) pays off in three places:

- **Ownership.** Each Blueprint component has one defaults file. Changing the ProductCard badge color touches `blueprint/components/product-card/default-props.tsx` — nothing else. Git history per file is narrow and meaningful.
- **Dead-code elimination.** When Stronger stops using a component, you remove the import and the entry in the aggregate. The default-props file is either deleted or orphaned — a trivial lint check.
- **Per-feature bundling.** Some clients code-split the aggregate per route, lazy-loading defaults for pages that use them. One file per component makes that trivial.

Miss Mary aggregates 42 defaults files. Stronger aggregates 59. Dedicated aggregates 43. Same structure, different component sets — each brand uses the Blueprint components it needs.

### When to add a new default

You add a new entry to the aggregate whenever you start using a new Blueprint component on a client. The flow is:

1. Create `app/project/blueprint/components/<new-component>/default-props.tsx` with the typed defaults.
2. Add an import + entry in `app/project/default-props.tsx`.
3. Rebuild.

That's it. The component registered under that name via `withManagedProps` will pick up the defaults automatically on its next render.

## 6. The GrowthBook integration

`src/frontend/grease/growthbook/src/grease-growthbook-managed-props-provider.tsx` is the bridge. It reads active GrowthBook features, turns flag names like `ProductCard::badges_location` into a managed-props tree, and passes it to `ManagedPropsProvider` as `deferredOverrideProps`:

```tsx
export const GreaseGrowthbookManagedPropsProvider = (
  props: GreaseGrowthbookManagedPropsProviderProps,
) => {
  const activeFeatures = useGreaseGrowthbookActiveFeatures();

  const [growthbookDeferredProps, setGrowthbookDeferredProps] =
    useState<ManagedDeferredPropsLiteral>(
      (props.deferredComponentNames ?? []).reduce((acc, name) => {
        acc[name] = null;
        return acc;
      }, {} as ManagedDeferredPropsLiteral),
    );

  const componentNameDelimiter = props.componentNameDelimiter ?? "::";
  const propDelimiter = props.propDelimiter ?? "_";

  useEffect(() => {
    if (activeFeatures?.length) {
      const deferredProps = resolveGrowthbookManagedProps(activeFeatures, {
        componentNameDelimiter,
        propDelimiter,
      });
      setGrowthbookDeferredProps(deferredProps);
    }
  }, [activeFeatures]);

  const growthbookManagedProps: ManagedPropsContextValue = {
    fallbackProps: props.managedProps?.fallbackProps,
    overrideProps: props.managedProps?.overrideProps,
    deferredOverrideProps: growthbookDeferredProps,
  };

  return (
    <ManagedPropsProvider value={growthbookManagedProps}>
      {props.children}
    </ManagedPropsProvider>
  );
};
```

Two pieces to notice:

1. **`deferredComponentNames` defers rendering.** Any component name listed here starts with a `null` entry. The `withManagedProps` HOC sees `null` and returns `null` — the component does not render until GrowthBook's active features are resolved. That prevents the "flag value arrives a tick late, page re-shapes" flicker on first paint.
2. **`resolveGrowthbookManagedProps`** (`src/frontend/grease/growthbook/src/utils/resolve-growthbook-managed-props.ts`) filters features by `::`, splits them into `[componentName, flatProps]`, and un-flattens the `_`-delimited path back into a nested object. So `ProductCard::badges_discountBadge_variants_color = "blue"` arrives at the component as `{ badges: { discountBadge: { variants: { color: "blue" } } } }`.

Clients compose both sources together. From `client-frontends/stronger/app/services/growthbook/stronger-growthbook-provider.tsx`:

```tsx
<GreaseGrowthbookProvider
  clientProviderProps={{ client, viewedExperiments: props.gbState?.e }}
  managedProps={{ fallbackProps: defaultProps }}
>
  {props.children}
</GreaseGrowthbookProvider>
```

`defaultProps` is the hand-written per-client theming. GrowthBook layers on top via `deferredOverrideProps`. Explicit props from pages sit in between. See [growthbook](./growthbook) for the surrounding provider setup.

## 7. Delimiters

Two delimiters, both overridable on `GreaseGrowthbookManagedPropsProvider`:

| Delimiter | Default | Role |
|---|---|---|
| `componentNameDelimiter` | `::` | Splits the flag key into `[ComponentName, flattenedPropPath]` |
| `propDelimiter` | `_` | Splits the prop path into nested keys |

So `ProductCard::badges_location` targets `ProductCard` and sets `badges.location`.

Change the defaults if your component names contain `::` (unlikely) or — more commonly — if your prop names contain underscores. A prop called `image_url` gets incorrectly split into `image.url` under the default `_` delimiter. Either rename the prop to `imageUrl` (preferred, follows repo convention) or switch `propDelimiter` to something your flag keys don't use.

## 8. When not to use it

Managed-props is for **runtime-dynamic** overrides: feature flags, A/B experiments, tenant variations that live outside the code. It has a non-trivial cost — every registered component runs through `useContext`, a flatten, a merge, and an un-flatten on every render.

Don't use it for:

- **Static theming.** Colours, spacing, fonts per client belong in `tailwind-theme.ts`. Tailwind classes compile to CSS; managed-props re-runs on every render.
- **One-off prop variations.** If a page needs a specific `ProductCard` shape once, pass explicit props. Don't register a global override.
- **Component selection.** Use template prop variants (`template: "1"` vs `template: "keen"`) or routing. Don't swap components via managed-props.

Use it when:

- The value depends on the tenant / environment and is known at bootstrap (`fallbackProps`).
- The value depends on a feature flag or experiment resolved at runtime (`deferredOverrideProps`).
- A CMS or BFF response decides a component's shape (`overrideProps`, or flatten into the tree via `withDelimitedProps`).

## Gotchas

- **`displayName` is load-bearing.** `withManagedProps` sets `Component.displayName = displayName` and uses it as the key in every lookup. Two components registered under the same name will collide silently.
- **Stale overrides.** `GreaseGrowthbookManagedPropsProvider` only re-resolves in a `useEffect` keyed on `activeFeatures`. If your GrowthBook client doesn't update its active-features list on flag change, overrides stick. Check that the active-features provider is wired up and streaming.
- **`null` deferred value hides the component.** Listing a name in `deferredComponentNames` returns `null` from the HOC until resolved — intentional, but surprising when a page renders blank and you didn't realise a flag pipeline was pending.
- **Underscores in prop names.** Default `propDelimiter` is `_`. `image_aspectRatio` becomes `image.aspectRatio`. Rename to camelCase or switch the delimiter.
- **React elements and dates survive flattening.** Everything else gets flattened and re-merged. If you're passing something exotic (class instances, functions other than getters), it'll be spread into keys and re-assembled — usually not what you want.
- **Object-tree vs function-getter values.** A prop value can be a plain object or a zero-arg function returning an object. Getters are called fresh on every render — handy for values that depend on time of day, but easy to cause render loops if the getter isn't stable.

## Troubleshooting

**Override not applied** — first check the component name matches exactly. `withManagedProps("ProductCard", ...)` only responds to `ProductCard`, not `product-card` or `ProductCardTemplate1`. Open React DevTools and confirm the wrapper is named `ManagedProps(ProductCard)`.

**GrowthBook flag has no effect** — check the delimiter. The key must be `ComponentName::prop_path`. A key without `::` is silently ignored by `resolveGrowthbookManagedProps`.

**Component renders blank** — check `deferredComponentNames`. If the name is listed but GrowthBook hasn't resolved yet, the HOC returns `null`. Either remove from the deferred list or verify the active-features pipeline fires.

**Prop path lands in the wrong nested key** — your prop name probably contains `_`. Switch `propDelimiter` or rename the prop.

**"Invalid ManagedPropsPropObject" warning in the console** — a non-object, non-getter value was supplied for a component entry. `resolveManagedProps` warns and continues; the component renders with whatever was mergeable.

**Zod validation error at runtime** — a managed-props override produced a shape the component's schema rejects. This is the system working as intended: `withZod` inside `withManagedProps` is the safety net. Fix the override or widen the schema.
