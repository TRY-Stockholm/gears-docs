## What you'll learn

Where to plug new integrations in, how the right URLs end up in your code, and which route owns each page. Every other guide assumes this.

## The three things to know

A Gears client is built on three conventions:

1. **One provider wraps everything.** `GearsConfigurationProvider` sits at the root. Any hook below it can ask for the current service URL or config value.
2. **The bootstrap file stacks providers in a specific order.** Configuration on the outside, data providers in the middle, feature integrations on the inside.
3. **Pages live under `app/gears/routes/`.** Each route receives a typed `frontendContext` (market, country, currency, etc.) from the server loader.

## 1. Configuration at the root

`GearsConfigurationProvider` carries the tenant and environment down to every hook.

```tsx
<GearsConfigurationProvider gearsConfiguration={rootLoaderData.gearsConfiguration}>
  {children}
</GearsConfigurationProvider>
```

`rootLoaderData.gearsConfiguration` comes from `gearsRootLoader` in `@made-people/gears-remix`, which reads env vars on the server and ships them to the client.

Two hooks are all you usually need:

```tsx
const bffUrl = useGearsServiceUrl("bff");
const tenantId = useGearsConfigurationVariable("tenantId");
```

Valid service keys: `"bff" | "engage" | "elevate" | "algolia" | "yotpo"`.

> **Never hard-code service hostnames.** Hard-coding bypasses multi-tenant deploys, environment switching, and the `localServices` override used for local dev.

## 2. The provider stack

The render tree looks like this:

```
GearsConfigurationProvider          ← outermost, required
  ├─ Infrastructure (images, links, translations)
  ├─ Data providers (Centra cart, account, search)
  └─ Feature integrations (Voyado, GTM, Ingrid, GrowthBook)
      └─ children
```

Order matters because inner providers read values from outer ones. For example, `VoyadoEngageProvider` reads the cart reference from Centra's token handler, so it must sit **inside** `CentraBootstrap`.

Real examples: `client-frontends/stronger/app/project/bootstrap.tsx` and `client-frontends/dedicated/app/project/bootstrap/bootstrap.tsx`.

## 3. Pages under `app/gears/routes/`

The root route (`app/root.tsx`) runs `gearsRootLoader` to fetch shared config. Page routes under `app/gears/routes/` receive `{ page, frontendContext }` from a wildcard route powered by `gearsLoader`.

Typical files you'll see:

| File | Owns |
|---|---|
| `centra-product-page-route.tsx` | PDP — mounts `ProductProvider` + product template |
| `storyblok-page-route.tsx` | CMS pages and category landings |
| `checkout-index-route.tsx` | Cart + checkout (`CentraCheckout`) |
| `checkout-finalize-route.tsx` | Post-payment confirmation |
| `search-page-route.tsx` | Algolia / Depict search results |
| `account/*` | Login, reset password, orders, profile |

Inside a route, only reach into gears config for URLs and tenant values. Everything else arrives as props.

## `frontendContext` at a glance

Each page receives a `FrontendContextV1` object — the per-request store context. Core fields:

- `market` — numeric Centra market id (as a string)
- `marketName` — human code like `"SE"` (optional, often defaulted)
- `country`, `language`, `currency`, `pricelist`
- `baseUrl`, `siteSlug`
- Optional `campaignSite`

Read it via route props — never from `window`.

## Directory cheat sheet

- `app/project/` — wiring: bootstrap, mappers, client-specific components
- `app/gears/routes/` — page components dispatched by the wildcard router
- `app/services/` — optional, client-specific driver setup (e.g. Stronger's GrowthBook driver)
- `app/root.tsx` — where the root loader lives

## Gotchas

- **Wrong provider order.** If `VoyadoEngageProvider` sits outside `CentraBootstrap`, `cartReference` is empty. If `AccountProvider` sits outside the Voyado driver, identify flows silently skip.
- **Missing `GearsConfigurationProvider`.** `useGearsServiceUrl` throws `"useGearsConfiguration must be used within a GearsConfigurationProvider"`.
- **Hard-coded URLs.** Silently hit the wrong cluster in preview or local dev.
- **`market` vs `marketName`.** `market` is a numeric id string. `marketName` is the code like `"SE"`.
- **Root vs page loader data.** `gearsConfiguration` comes from the root loader; `frontendContext` comes from the page loader. Mixing them up surfaces as `undefined` props.

## Troubleshooting

**"X must be used within Y"** — a hook is mounted above its provider. Check the stack order in `bootstrap.tsx`.

**Service URL returns the wrong tenant or cluster** — an env var on the server is unset. Inspect `rootLoaderData.gearsConfiguration`; any `undefined` field points to a missing env var.

**Page renders blank, route loader returns `null`** — `gearsRootLoader` skipped this URL. Check `shouldHandleUrl` and confirm the BFF returned a `settingsAndContentResponse`.
